## 2026-06-09 · v3.12.0：PWA 分享目标（4.0 支柱 2）——相册分享直达印迹

> 4.0「成组与流程」三条支柱里，批量加水印（v3.8.0）和风格包（v3.6.0 + v3.7.0）已提前落地，只剩支柱 2「PWA 分享目标」没做。本版把它收尾。

### 实现

- **manifest.webmanifest**：加 `share_target`——`action:"./share-target"`、`method:"POST"`、`enctype:"multipart/form-data"`、接收 `photo` 图片文件。
- **sw.js**（CACHE → `yinji-v53`）：
  - **POST 拦截**：`fetch` listener 新增 POST `./share-target` 分支 → `e.request.formData()` 取 `photo` 文件 → 写入专用缓存 `yinji-shared`（key=`shared-image`，Response header 带 `X-Shared-File-Name` 存原文件名）→ `Response.redirect("./watermark.html?shared=1")`。失败兜底→不带参重定向。
  - **GET 回传**：新增 GET `./__shared_image` 分支 → 从 `yinji-shared` 取 `shared-image` → 返回给页面 → 立即 `cache.delete`（用完即删）。无图返回 404。
  - **缓存隔离**：`activate` 清理时排除 `SHARED_CACHE`，防止 SW 更新时误删正在传递的分享图片。
- **watermark.html**（`APP_VERSION=v3.12.0`）：
  - 启动流程加一步：`navigator.serviceWorker.ready` 后检查 `location.search` 含 `shared=1` → `history.replaceState` 清 URL 参数 → `fetch("./__shared_image")` → 从 response header 取 `X-Shared-File-Name` 还原文件名 → `resp.blob()` + `new File()` → 走既有 `loadFile(file)`。
  - **全复用既有的照片加载管线**：`loadFile` 负责 EXIF 解析 / HEIC 检测 / bitmap 解码 / setMode / draw，零新渲染代码。

### 关键设计决策

- **离线码暂存 + fetch 回传**，而非 postMessage 传 Blob（SW→Client postMessage 跨上下文传二进制兼容性差）。
- **iOS Safari Web Share Target 支持历来弱**——诚实标注：安卓 / 桌面 Chrome 体验好，iOS 可能不出现入口，**降级 = 照旧手动选图，不影响任何现有功能**。
- 专用缓存 `yinji-shared`（独立于主 `CACHE`），防止 SW 更新清理时误删 + 不与正常离线缓存混在一起。

### 收尾

- 自检：`node --check`=SYNTAX_OK（watermark.html 内联 JS + sw.js 均过）；加载期外链=LINT_OK；manifest JSON_OK。
- `APP_VERSION` → `v3.12.0`；`sw.js` CACHE → `yinji-v53-2026-06-09-v3.12.0`。

## 2026-06-09 · v3.9.0：液态玻璃强度可调（关/轻/标准/强）+ 更接近 iOS 26/27 Liquid Glass

> 南心"适配 iOS27 液态玻璃效果可调"。现有玻璃已是变量驱动(`--glass-blur`/`--glass-blur-sm` + 饱和 + brightness + 高光鳞边),且主玻璃面(面板/弹层/芯片/输入框)都读这俩变量。故**可调=用 CSS 变量覆盖强度,不逐个改规则**;**iOS27 适配=强档更透更亮的高光鳞边**。改动小、零渲染管线影响、纯本地。

### 实现

- **CSS 分档(变量驱动)**:新增 `body[data-glass-level="light"]` / `["strong"]` 覆盖 `--glass-blur`/`--glass-blur-sm`/`--glass-highlight`/`--glass-border`。**标准 = `:root` 默认值(不另写 block,保持老用户默认观感不变)**。轻=blur 16/10px、低饱和;强=blur 64/42px、saturate 210%、更亮高光鳞边(`inset 0 1.5px 0 rgba(255,255,255,.72)`)+ 更亮 border(rgba 白 .46)。**只覆盖模糊/饱和/鳞边(主题无关),不动 `--glass-bg` 透明度**(避免与深色主题的 alpha 覆盖打架)。`data-glass="off"` 仍走原纯色路径。
- **JS 改 on/off → 分档**:`GLASS_KEY="yinji.glass"` 存 level 字符串;`normGlassLevel` **兼容老存档**("0"→off、"1"/null→standard、非法→standard);`applyGlass(level)`:off → `data-glass="off"`+移除 `data-glass-level`,否则 `data-glass="on"`+`data-glass-level=level`,写回 localStorage;`syncGlassSeg` 切 chip `.on`。自包含(不依赖 segBind/segSelect 的定义顺序)。`syncSettingsUI` 里原来 `glassToggle.checked=…` 一行改成 `syncGlassSeg(…)`(否则删了 `var glassToggle` 会 ReferenceError)。
- **HTML**:设置「外观」里玻璃从 `.switch` 复选框换成 `.seg#glassSeg`(关/轻/标准/强,data-g),与上方「主题」分段控件同款。

### 收尾

- `APP_VERSION` → `v3.9.0`；`sw.js` CACHE → `yinji-v50-2026-06-09-v3.9.0`。
- 自检:`node --check`=SYNTAX_OK;`glassToggle` 残留=0、`#glassSeg`/`data-glass-level` 两档/`applyGlass`/`syncGlassSeg`/`normGlassLevel` 全在;加载期外链=NONE。
- 浏览器验证(清 `yinji.glass`):默认=标准(`--glass-blur`=blur44 / data-glass-level=standard / seg 高亮标准 / localStorage 写回 standard);四档点击 → body `--glass-blur` 逐档变(none / 16 / 44 / 64)+ data 属性正确 + 持久化;**真实 `.panel` 计算后 backdrop-filter 逐档生效**(关 none / 标准 blur(44px) saturate(1.8) / 强 blur(64px) saturate(2.1));设置面板打开不报错 + seg 同步;**老存档 "0" → off 平滑迁移**(data-glass=off / seg 高亮关 / 规范化写回 "off");console 零报错。

## 2026-06-09 · v3.8.0：批量加水印（4.0「成组与流程」头牌 · 复用单张管线逐张跑 + 零依赖 store-zip）

## 2026-05-30 · v2.8.0：玻璃磨砂做实 + 状态栏统一 + 复制到剪贴板 / 记住选择 / 长按看原图

### 修复 1：玻璃磨砂重做（downscale-upscale 取代 `ctx.filter="blur()"`）

- **根因**：v2.7.1 玻璃款用 `ctx.filter="blur(20px)"` 给 canvas 区域磨砂，但 **iOS Safari 的 canvas 2D `ctx.filter` 对 `blur()` 支持极弱 / 几乎不生效**（桌面 Chrome 正常）——真机上看着跟没模糊一样。
- **新方案**：缩小再放大的「像素抹平」式磨砂，所有浏览器都work：
  - 新增 `frostDraw(x,y,ww,hh)`：把目标区域 `drawImage` 到一块极小的离屏 canvas（宽 `bw=max(16, round(ww/24))`，高按比例），再开 `imageSmoothingEnabled=true` + `imageSmoothingQuality="high"` 平滑放大回原尺寸——降采样丢高频细节 = 等效强模糊。放大时叠 `ctx.filter="saturate(155%) brightness(1.06)"`（有就用、`typeof` 守卫），让磨砂区更有「液态玻璃」的提亮通透感。
  - 新增 `frostRect(x,y,ww,hh)`（矩形 clip）和 `frostRound(x,y,ww,hh,r)`（圆角 clip，复用既有 `roundRectPath`）两个包装，`save→clip→frostDraw→restore`。
  - `drawGlassBar()`：旧的 `if(typeof ctx.filter...)` blur 块替换成 `frostRect(0, barTop, w, barH)`，tint 加深到 `rgba(20,20,24,.26→.44)`、高光线 `rgba(255,255,255,.42)`、厚度 `round(h*0.0018)`。
  - `drawGlassCard()`：blur 块替换成 `frostRound(chipX, chipY, chipW, chipH, r)`。
- **降级**：离屏 canvas + `drawImage` 缩放是 baseline 能力，无需守卫；`ctx.filter` 仅用于锦上添花的提亮，`typeof` 守卫后老浏览器跳过、磨砂本体仍在。

### 修复 2：删除底部「添加到主屏幕」提示

- 移除编辑区底部的 `<p class="hint">想当成 App 用？…添加到主屏幕</p>`。装桌面入口仍在浏览器原生菜单，不影响 PWA 安装；界面更干净。

  - 高清 JPEG：`toDataURL("image/jpeg",0.98)`，画质更高、文件略大。
  - 无损 PNG：`toDataURL("image/png")`，文件最大、**不含 EXIF**。
- **`buildExport()`**：按画质产出 `{blob, mime, ext}`。JPEG 路径在生成后若有 `state.exifApp1` 则调 `spliceApp1` 把拍摄信息拼回去；PNG 直接返回。
- **`setExportQuality(q)`**：校验入参、写 `state.exportQuality`、切换三个按钮的 `.on` 状态，并动态刷新说明文字（PNG=「PNG · 无损画质，文件最大，不含拍摄信息」；JPEG=高清?「更高画质，文件略大」:「体积小，日常分享够用」，且**仅当** `state.exifApp1` 存在时才追加「，保留拍摄信息」——不空许诺）。

### 保存到相册（Web Share API）

- `#saveAlbumBtn` 处理函数：`buildExport()` → 包成 `File` → `if(navigator.canShare && navigator.canShare({files:[file]}) && navigator.share)` 走 `navigator.share({files:[file]})`，iOS 上即「存储图像」直接进相册；不支持时退化 `downloadBlob`。
- **iOS 用户激活陷阱**：`share()` 必须在点击手势的**同步调用栈**里触发。若用 `setTimeout` 延迟（哪怕 0ms）会丢失 user-activation，分享面板打不开。所以 PNG 编码的短暂卡顿放在同步栈里，不延迟。
- `share()` 的 `catch`：`AbortError`（用户取消）静默忽略，其它错误降级为 `downloadBlob`。
- 不传 `title`：保持 iOS 规范的「存储图像」入口，避免被识别成「分享给某人」。

### 保留拍摄信息（EXIF 回写）

导出 JPEG 时把原图的拍摄信息拼回去（PNG 不支持 EXIF，故无）。新增一组 helper：

- **`extractExifApp1(buffer)`**：分发器。源是 JPEG（`FFD8`）→ `extractJpegApp1`；源是 HEIC（偏移 4 是 `ftyp`）→ `buildHeicApp1`。`loadFile` 里 try/catch 调用，失败置 `state.exifApp1=null`。
- **`extractJpegApp1(view)`**：遍历 APP 段，找到 `FFE1` 且数据以 `Exif` 开头的段，原样拷出 `2+segLen` 字节（完整保真，含 GPS 等所有标签），调 `neutralizeOrientation`。
- **`buildHeicApp1(view)`**：复用 `locateHeicExif` 定位内嵌 TIFF，`tiffEnd=exifOffset+itemLen`，组装 `[FFE1][segLen][Exif\0\0][TIFF]`（`segLen=2+6+tiffLen`，须 ≤0xFFFF），TIFF 拷到 index 10，调 `neutralizeOrientation`。
- **`neutralizeOrientation(app1)`**：TIFF 在 app1 的 index 10，读字节序，在 IFD0 找标签 `0x0112`，`dv.setUint16(e+8, 1, little)` 强制写 1。
  - **方向陷阱**：`createImageBitmap(file,{imageOrientation:"from-image"})` 已经把像素摆正，导出的 canvas 像素是正的。如果再把原图的 `Orientation`（可能是 6/8 等）原样拼回去，查看器会**二次旋转**。所以必须把回写的 EXIF Orientation 抹成 1。
- **`spliceApp1(jpegBytes, app1)`**：把 app1 插到 JPEG `SOI`（`FFD8`）之后：`out[0]=0xFF;out[1]=0xD8;out.set(app1,2);out.set(jpegBytes.subarray(2),2+app1.length)`。
- **`dataURLToUint8(dataURL)`**：`atob` 解码逗号后的 base64 成 `Uint8Array`，供拼接用。
- **`parseHeicExif` 重构**：抽出 `locateHeicExif(view)`（返回 `{tiffStart, exifOffset, itemLen}`），`parseHeicExif` 变薄壳：`var loc=locateHeicExif(view); if(!loc) return null; return readTiff(view, loc.tiffStart);`。解析与回写共用同一套盒子遍历逻辑，不重复。

### v2.5.0：HEIC 拍摄信息读取

#### HEIC EXIF 解析（单文件零依赖）

HEIC 是 ISOBMFF 容器（和 MP4 同结构），EXIF 存在 `meta` 盒子的 item 里。完整解析步骤：

1. **判断格式**：`ftyp` 盒子（文件偏移 4）的 major brand 或 compatible_brands 是 HEIC 系（heic / heix / hevc / mif1 / avif / heif...）。
2. **找 meta 盒子**（top-level，FullBox）。
3. **遍历 iinf → infe** 找 `item_type == 'Exif'`（`0x45786966`）的 item，记录 item_id。
4. **遍历 iloc** 找该 item_id 的 (base_offset + extent_offset) + length。
5. **读取 item data**：头 4 字节是 `exif_tiff_header_offset`（大端 u32，通常是 6，= "Exif\0\0" 长度），跳过它再跳过对应字节，剩下的就是标准 TIFF 数据。
6. **复用既有 `readTiff`** 把 TIFF 解析成 tags 对象，和 JPEG 路径输出格式一致。

代码组织：

- `parseExif(buffer)` 改成分发器：头两字节是 `FF D8` 走 `parseJpegExif`；偏移 4 是 `ftyp` 走 `parseHeicExif`。
- 新增 `parseHeicExif`、`isHeicLike`、`isHeicBrand`、`findBox`、`readVarSize` 五个函数。
- `parseHeicExif` 处理 infe v2 / v3、iloc v0 / v1 / v2 三种版本（覆盖所有现代 HEIC 文件）。

> v2.7.0 把 `parseHeicExif` 进一步拆出 `locateHeicExif`，与导出回写共用定位逻辑。

#### 图像渲染的限制

HEIC **图像** 解码只在 iOS Safari / macOS Safari 上工作（浏览器原生）。Chrome / Edge / Firefox 没有 HEIC 解码器，要装 libheif-js（700KB+ WASM）才能解，违反「单文件零依赖」约束，**本轮不做**。

体面降级：

- `loadFile` 加 `isHeic` 标记（file 名后缀 `.heic/.heif/.avif` 或 magic bytes 命中）。
- `loadBitmap` 失败时，如果 `isHeic` 为真，抛出「HEIC 格式，当前浏览器解不开。请用 iPhone Safari 打开，或先转成 JPEG。」，在 drop 区显示。

## 备注

- 关键约束：**单文件 + 零外部依赖**（外链资源会导致手机白屏）。
- **路线已收敛：统一走 HTML / PWA**，不做安卓原生（Capacitor 路线已于 v2.3.0 弃用、v2.7.0 起相关文件全部移除）、不做 iOS 原生。Win .exe（Tauri）留可选，优先级低。
- 「样式 vs 文字模式」正交设计：加新样式不重复实现手动 / 自动两套文字逻辑。
- 模板登记表 `TEMPLATES` + `DRAW_DISPATCH` + `customTemplates`：所有样式必须登记。
- **导出**：`buildExport()` 按 `state.exportQuality` 产图；JPEG 走 `spliceApp1` 回写 `state.exifApp1`（PNG 无 EXIF）；保存到相册走 `navigator.share({files})`，必须同步调用（iOS 用户激活）。回写 EXIF 必须把 Orientation 抹成 1（像素已被 createImageBitmap 摆正）。
- 日志：**两份中文名文档**——`更新日志-用户版.md`（每天一条 App Store 风格总结）+ `更新日志-开发版.md`（按版本倒序的详细技术变更），都带日期。根目录 `CHANGELOG.md` 是 SemVer 用户视角条目。
- 「md + docx」双格式：改完任何 .md，交付前跑 `python tools/md2docx.py` 重新生成同名 .docx（脚本递归扫 docs/** 和 devlog/**）。
- 版本号管理：`APP_VERSION` 常量是单一真相源，CHANGELOG / devlog / sw.js cache 名称 / UI 显示都跟它走。
- PWA 部署必须**多文件 HTTPS**（GitHub Pages / Netlify / Cloudflare Pages），DevFile.cn 单文件不行。
