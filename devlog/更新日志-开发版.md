## 2026-06-01 · v2.12.0：2.0 收官版 —— 调色（色调影调 Beta）+ 玻璃四周模板 + 选择模板改名

> 2.x 大版本收官。三件主功能 + 一处改名 + 小修。核心思路:**调色在 `getSize()` 单点注入**,35 款样式 + 导出全部自动继承,零逐样式改动。仍单文件 + 零外链;`node --check` = SYNTAX_OK、无 `http(s)://` / CDN / 外链字体。3.0 起主打 AI,并把本版 Beta 调色做成完整版。

### 调色 · 色调影调（Beta）—— getSize 单点注入 + 像素变换 + 缓存

- **架构**:照片只调一次色,结果存进离屏 canvas 缓存;`getSize()` 返回缓存 canvas 当 `bmp` → 所有样式的 `ctx.drawImage(s.bmp,…)` 和导出(`buildExport` 读已绘制的 `canvas`)**自动吃到调色**,无需改任何 draw 函数。
- **为什么手动改像素而非 `ctx.filter`**:iOS Safari 对 canvas 不应用 `ctx.filter` 的颜色类滤镜(saturate/contrast 等),实测无效 → 改用 `getImageData` / `putImageData` 逐像素算。
- `state.tone`(默认 `"none"`);`TONE_LIST` 8 项(none 原图 / bw 黑白 / film 复古 / warm 暖阳 / cool 冷调 / fresh 清新 / gray 高级灰 / vivid 鲜艳);`TONE_PARAMS` 每款一组 `{sat,con,bri,temp,tint,lift}`。
- `applyTone(d,p)`:逐像素按 **饱和度 → 色温/色调(temp/tint)→ 对比度 → 亮度 → 提亮暗部(lift)** 顺序算,全程 clamp 0–255。
- `gradedBitmap(bmp,w,h)`:`tone==="none"` 直接返原图;否则按 `tone/w/h/src` 做缓存键,命中复用、未命中重算;`getImageData` 包 try/catch,污染画布(tainted)时回退原图不报错。
- `invalidateGrade()`:换图 / 重置时清缓存(reset 处 `state.bitmap=null` 后补调一次)。
- **长按看原图**`drawOriginalOnly` 改 `ctx.drawImage(state.bitmap,…)`——画真·原片(不调色、不水印),对照更直观。
- **UI**:样式条下方加 `.tone-block`(「调色 · 色调影调」标签 + `.beta-tag` Beta 角标)+ `.tone-seg` chip 行;`renderToneSeg()` / `setTone(tone)`;选中态高亮。
- **持久**:`savePrefs` 写 `tone:state.tone`;`initPrefs` 校验 `p.tone==="none"||TONE_PARAMS[p.tone]` 后还原;IIFE 后调一次 `renderToneSeg()`。

### 玻璃四周（glassFrame）—— 复用 frostRect 的四边磨砂相框

- `drawGlassFrame()`:四边各一条 `frostRect` 磨砂边(边宽 `m=max(22, min(w,h)*0.052)`),叠 `rgba(248,250,252,.16)` 提亮;底栏 `barH=max(76,h*0.14)`(封顶 `h*0.4`)叠上→下加深的渐变放文字;中间照片区描 `rgba(255,255,255,.55)` 细内框 + 顶部一条高光线。文字走 `drawText` 白字配置。
- 登记三处:`DRAW_DISPATCH.glassFrame`、`TEMPLATES`(`{id:"glassFrame",name:"玻璃四周",…}`)、`CAT_IDS.glass` 加 `glassFrame`(与 glassBar / glassCard 同「玻璃」分类)。

### 添加模板 → 选择模板（改名 + 改行为）

- 顶部大按钮文案 `选择模板`(title / aria-label 同步);点击从 `openCustomEditor(null)` 改为新抽出的 `openTemplatePicker()`(打开「更多模板」面板)。

## 2026-05-31 · v2.11.0：PWA「有新版本」更新提示（updatefound → 提示条 → skipWaiting + reload）

> roadmap §3.4。让已"装到主屏"的用户在新版上线后收到提示、一键刷新到新版，取代过去"靠 bump cache 名、用户手动重开才生效"的弱感知更新。纯 PWA / 外壳逻辑，不碰 Canvas 渲染。仍单文件 + 零外链；`node --check` = SYNTAX_OK、无外部资源。

### sw.js：更新策略从"激进自动"改成"等用户点"

- `install` 去掉 `self.skipWaiting()`：新版改停在 `waiting`，不再抢着激活，避免用户正在编辑时被强制切换。
- 新增 `message` 监听：`if(e.data && e.data.type==='SKIP_WAITING') self.skipWaiting();`，仅当页面「刷新」按钮发来指令才激活。
- `activate` 的 `clients.claim()` 保留：首次安装（无旧 sw 控制页面）时浏览器本就直接 activate，claim 让其当次接管、离线即时可用。
- cache 名 → `yinji-v18-2026-05-31-v2.11.0`。

### watermark.html：检测 + 提示条 + 一键刷新

- PWA 注册段重写：`register('sw.js').then(reg => …)`。
  - 进页面即 `reg.waiting && navigator.serviceWorker.controller` → `showUpdateBar(reg.waiting)`（有 controller 才算"更新"而非首装）。
  - `reg.onupdatefound` → `reg.installing` 的 `statechange`，到 `installed && controller` → `showUpdateBar(nw)`。
- `controllerchange` → `location.reload()`，但前置 **`swReloading` 闸**：仅用户点过「刷新」（按钮里先置 `swReloading=true` 再 `postMessage`）才重载。否则首装时 `clients.claim()` 触发的 controllerchange 会误刷新一次——这是关键防护。
- UI：底部液态玻璃 `.update-bar`（`--glass-bg-strong` + `--glass-blur-sm`，深色 / 玻璃关闭模式靠 CSS 变量自动适配），含「刷新」(`--accent` 实心) + 「×」关闭；`role=status` `aria-live=polite`。`showUpdateBar` / `hideUpdateBar` 切 `.show` 类。
- 「刷新」点击：`swReloading=true; waitingSW.postMessage({type:'SKIP_WAITING'})`；「×」只隐藏、不更新。

### 已知特性（非 bug）

- 本版上线后**老用户当次不弹提示**——他们正跑的旧 sw 没有这套检测逻辑，需手动刷新一次拿到 v2.11.0；**从 v2.11.0 起**再发新版才会触发提示。这是 SW 更新模型固有的"隔一版生效"。

- `applyDefaultText()`：按 `defaultText` 预填手动文字框（`date` 今天日期 / `blank` 留空 / `fixed` 用 `fixedText`）。取代旧的日期预填 IIFE。

- `buildInfo(ex)`：取 `af=ensureSettings().autoFields`，六个字段逐一门控（`af.device/time/aperture/shutter/iso/focal`）。作为自动模式字段的**唯一收口**，屏显面板 + 所有底栏样式都一处生效。

## 2026-05-30 · v2.8.0：玻璃磨砂做实 + 状态栏统一 + 复制到剪贴板 / 记住选择 / 长按看原图

> 本版含 3 个真机反馈修复（玻璃没模糊 / 底部多余提示 / 状态栏色不统一）+ 路线图里挑的 3 个新功能（复制图片、记住选择、长按看原图）。

### 修复 1：玻璃磨砂重做（downscale-upscale 取代 `ctx.filter="blur()"`）

- **根因**：v2.7.1 玻璃款用 `ctx.filter="blur(20px)"` 给 canvas 区域磨砂，但 **iOS Safari 的 canvas 2D `ctx.filter` 对 `blur()` 支持极弱 / 几乎不生效**（桌面 Chrome 正常）——真机上看着跟没模糊一样。
- **新方案**：缩小再放大的「像素抹平」式磨砂，所有浏览器都work：
  - 新增 `frostDraw(x,y,ww,hh)`：把目标区域 `drawImage` 到一块极小的离屏 canvas（宽 `bw=max(16, round(ww/24))`，高按比例），再开 `imageSmoothingEnabled=true` + `imageSmoothingQuality="high"` 平滑放大回原尺寸——降采样丢高频细节 = 等效强模糊。放大时叠 `ctx.filter="saturate(155%) brightness(1.06)"`（有就用、`typeof` 守卫），让磨砂区更有「液态玻璃」的提亮通透感。
  - 新增 `frostRect(x,y,ww,hh)`（矩形 clip）和 `frostRound(x,y,ww,hh,r)`（圆角 clip，复用既有 `roundRectPath`）两个包装，`save→clip→frostDraw→restore`。
  - `drawGlassBar()`：旧的 `if(typeof ctx.filter...)` blur 块替换成 `frostRect(0, barTop, w, barH)`，tint 加深到 `rgba(20,20,24,.26→.44)`、高光线 `rgba(255,255,255,.42)`、厚度 `round(h*0.0018)`。
  - `drawGlassCard()`：blur 块替换成 `frostRound(chipX, chipY, chipW, chipH, r)`。
- **降级**：离屏 canvas + `drawImage` 缩放是 baseline 能力，无需守卫；`ctx.filter` 仅用于锦上添花的提亮，`typeof` 守卫后老浏览器跳过、磨砂本体仍在。

### 修复 2：删除底部「添加到主屏幕」提示

- 移除编辑区底部的 `<p class="hint">想当成 App 用？…添加到主屏幕</p>`。装桌面入口仍在浏览器原生菜单，不影响 PWA 安装；界面更干净。

### 修复 3：顶部状态栏颜色统一（`theme-color`）

- **根因**：`<meta name="theme-color">` 原为奶油色 `#EFEAE0`，但页面顶部那层 `radial-gradient(130% 90% at 50% -10%, rgba(255,255,255,.65) …)` 高光 + `background-attachment:fixed`（顶端永远接近纯白），导致手机状态栏色块和页面顶端「两张皮」。
- **改**：`theme-color` 调亮到 `#F4F0E8`，取奶油底色与顶端近白高光的折中值，状态栏与页面顶端基本融为一体。

### 新增 1：复制图片到剪贴板（`copyImgBtn`）

- 保存弹窗里 `saveAlbumBtn` 与 `.row` 之间加 `<button id="copyImgBtn" hidden>复制图片</button>`（默认 `hidden`）。
- **特性门控**：仅当 `window.ClipboardItem && navigator.clipboard && navigator.clipboard.write` 都在时才 `hidden=false`，否则按钮保持隐藏——老浏览器看不到、不报错。
- **同步手势**：点击里**同步**用 `dataURLToUint8(canvas.toDataURL("image/png"))` 造 Blob → `navigator.clipboard.write([new ClipboardItem({"image/png":blob})])`。和 `navigator.share` 同理，blob 必须在 click 手势里同步生成，否则 iOS 丢失 user activation。用 **PNG**（剪贴板图片支持面最广）。成功 `toast("图片已复制，可直接粘贴")`，失败回退提示改用「保存到相册」。

### 新增 2：记住上次的样式 / 模式 / 画质（`yinji.prefs.v1`）

"auto" : "manual"), true)`——记住的意图优先，且 `silent=true` 不反向覆盖 `prefMode`。

### 导出管线（核心）

以前保存只有一种 `toDataURL("image/jpeg",0.92)`，且默认落到「文件」而非相册。本轮重做导出：

### 保留拍摄信息（EXIF 回写）

导出 JPEG 时把原图的拍摄信息拼回去（PNG 不支持 EXIF，故无）。新增一组 helper：

- **`extractExifApp1(buffer)`**：分发器。源是 JPEG（`FFD8`）→ `extractJpegApp1`；源是 HEIC（偏移 4 是 `ftyp`）→ `buildHeicApp1`。`loadFile` 里 try/catch 调用，失败置 `state.exifApp1=null`。
- **`extractJpegApp1(view)`**：遍历 APP 段，找到 `FFE1` 且数据以 `Exif` 开头的段，原样拷出 `2+segLen` 字节（完整保真，含 GPS 等所有标签），调 `neutralizeOrientation`。
- **`buildHeicApp1(view)`**：复用 `locateHeicExif` 定位内嵌 TIFF，`tiffEnd=exifOffset+itemLen`，组装 `[FFE1][segLen][Exif\0\0][TIFF]`（`segLen=2+6+tiffLen`，须 ≤0xFFFF），TIFF 拷到 index 10，调 `neutralizeOrientation`。
- **`neutralizeOrientation(app1)`**：TIFF 在 app1 的 index 10，读字节序，在 IFD0 找标签 `0x0112`，`dv.setUint16(e+8, 1, little)` 强制写 1。
  - **方向陷阱**：`createImageBitmap(file,{imageOrientation:"from-image"})` 已经把像素摆正，导出的 canvas 像素是正的。如果再把原图的 `Orientation`（可能是 6/8 等）原样拼回去，查看器会**二次旋转**。所以必须把回写的 EXIF Orientation 抹成 1。
- **`spliceApp1(jpegBytes, app1)`**：把 app1 插到 JPEG `SOI`（`FFD8`）之后：`out[0]=0xFF;out[1]=0xD8;out.set(app1,2);out.set(jpegBytes.subarray(2),2+app1.length)`。
- **`dataURLToUint8(dataURL)`**：`atob` 解码逗号后的 base64 成 `Uint8Array`，供拼接用。
- **`parseHeicExif` 重构**：抽出 `locateHeicExif(view)`（返回 `{tiffStart, exifOffset, itemLen}`），`parseHeicExif` 变薄壳：`var loc=locateHeicExif(view); if(!loc) return null; return readTiff(view, loc.tiffStart);`。解析与回写共用同一套盒子遍历逻辑，不重复。

### v2.5.0：HEIC 拍摄信息读取

#### HEIC EXIF 解析（单文件零依赖）

HEIC 是 ISOBMFF 容器（和 MP4 同结构），EXIF 存在 `meta` 盒子的 item 里。完整解析步骤：

1. **判断格式**：`ftyp` 盒子（文件偏移 4）的 major brand 或 compatible_brands 是 HEIC 系（heic / heix / hevc / mif1 / avif / heif...）。
2. **找 meta 盒子**（top-level，FullBox）。
3. **遍历 iinf → infe** 找 `item_type == 'Exif'`（`0x45786966`）的 item，记录 item_id。
4. **遍历 iloc** 找该 item_id 的 (base_offset + extent_offset) + length。
5. **读取 item data**：头 4 字节是 `exif_tiff_header_offset`（大端 u32，通常是 6，= "Exif\0\0" 长度），跳过它再跳过对应字节，剩下的就是标准 TIFF 数据。
6. **复用既有 `readTiff`** 把 TIFF 解析成 tags 对象，和 JPEG 路径输出格式一致。

代码组织：

- `parseExif(buffer)` 改成分发器：头两字节是 `FF D8` 走 `parseJpegExif`；偏移 4 是 `ftyp` 走 `parseHeicExif`。
- 新增 `parseHeicExif`、`isHeicLike`、`isHeicBrand`、`findBox`、`readVarSize` 五个函数。
- `parseHeicExif` 处理 infe v2 / v3、iloc v0 / v1 / v2 三种版本（覆盖所有现代 HEIC 文件）。

> v2.7.0 把 `parseHeicExif` 进一步拆出 `locateHeicExif`，与导出回写共用定位逻辑。

#### 图像渲染的限制

HEIC **图像** 解码只在 iOS Safari / macOS Safari 上工作（浏览器原生）。Chrome / Edge / Firefox 没有 HEIC 解码器，要装 libheif-js（700KB+ WASM）才能解，违反「单文件零依赖」约束，**本轮不做**。

- `loadFile` 加 `isHeic` 标记（file 名后缀 `.heic/.heif/.avif` 或 magic bytes 命中）。

## 备注

- 关键约束：**单文件 + 零外部依赖**（外链资源会导致手机白屏）。
- **路线已收敛：统一走 HTML / PWA**，不做安卓原生（Capacitor 路线已于 v2.3.0 弃用、v2.7.0 起相关文件全部移除）、不做 iOS 原生。Win .exe（Tauri）留可选，优先级低。
- 「样式 vs 文字模式」正交设计：加新样式不重复实现手动 / 自动两套文字逻辑。
- 模板登记表 `TEMPLATES` + `DRAW_DISPATCH` + `customTemplates`：所有样式必须登记。
- **导出**：`buildExport()` 按 `state.exportQuality` 产图；JPEG 走 `spliceApp1` 回写 `state.exifApp1`（PNG 无 EXIF）；保存到相册走 `navigator.share({files})`，必须同步调用（iOS 用户激活）。回写 EXIF 必须把 Orientation 抹成 1（像素已被 createImageBitmap 摆正）。
- 日志：**两份中文名文档**——`更新日志-用户版.md`（每天一条 App Store 风格总结）+ `更新日志-开发版.md`（按版本倒序的详细技术变更），都带日期。根目录 `CHANGELOG.md` 是 SemVer 用户视角条目。
- 「md + docx」双格式：改完任何 .md，交付前跑 `python tools/md2docx.py` 重新生成同名 .docx（脚本递归扫 docs/** 和 devlog/**）。
- 版本号管理：`APP_VERSION` 常量是单一真相源，CHANGELOG / devlog / sw.js cache 名称 / UI 显示都跟它走。
- PWA 部署必须**多文件 HTTPS**（GitHub Pages / Netlify / Cloudflare Pages），DevFile.cn 单文件不行。
