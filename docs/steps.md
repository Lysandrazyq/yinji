# 执行步骤

> 记录怎么开发、怎么测、怎么发布、怎么和负责人协作。

## 1. 协作流程（和非程序员负责人）

1. **沟通需求**：用大白话讲清楚要做什么、有哪些取舍，**先确认，再动手**。
2. **开发**：按 `requirements.md` / `tech.md` / `design.md` 执行，不偏离既有标准。
3. **自检**：见下方第 3 节。
4. **交付**：给出文件 + 简明"怎么用"步骤。
5. **写日志**：更新 `devlog/` 下两份日志——`更新日志-用户版.md`（当天一条大白话总结）+ `更新日志-开发版.md`（最上方加一节详细技术变更），都带日期。

## 2. 改动 / 新增功能的步骤

1. 读 `CLAUDE.md` 和 `docs/` 下相关文件。
2. 在 `watermark.html` 上修改（保持单文件、零外部依赖）。
3. 自检通过后更新交付文件。
4. 如涉及需求 / 技术 / 设计变化，**同步更新对应的 docs 文件**。
5. 写当日开发日志。

## 3. 自检清单（每次改完必做）

1. **语法检查**：提取内联 `<script>` 内容，`node --check` 验证无语法错误。
2. **EXIF 验证**：用一张带拍摄信息的样张，确认设备 / 时间 / 参数解析正确。
3. **排版检查**：肉眼看水印不出框、不重叠、长文本能自适应。
4. **依赖检查**：确认文件内无任何 `http(s)://`、CDN、外链字体。

## 3.5 文档双格式（md + docx）

`docs/` 下每份文档都有两个版本：

- `.md`：工作格式，写 / 改在这里。
- `.docx`：给负责人 Word 打开看的，**由 .md 自动生成**，不要手动编辑（下次重新生成会覆盖）。

**改完 .md 后，AI 在交付前要重新生成 .docx**：

```bash
python tools/md2docx.py
```

脚本递归扫 `docs/**` 和 `devlog/**` 下所有 `.md`（含中文名的两份更新日志），输出同名 `.docx`。中文字体用 Microsoft YaHei，保留标题层级、列表、加粗、行内代码、表格、引用样式。

依赖：`pip install python-docx`（一次性）。

## 3.6 跨平台打包(可选,负责人想做的)

完整流程见 `docs/cross-platform.md`,这里只放速查:

- **PWA 部署**(立即可用):整个 `yinji/` 文件夹上传到 GitHub Pages / Netlify,用户浏览器打开网址就能"安装"。DevFile.cn 不行(单文件)。
- **Windows .exe**(可选):`npx tauri build`,产物在 `src-tauri/target/release/bundle/msi/`。
- 改 `watermark.html` 后:PWA 自动同步(重新部署即可);Tauri 需要重新 build。

> 安卓 Capacitor `.apk` 路线已于 v2.3.0 弃用,改为统一 PWA。

## 4. 部署 / 更新（DevFile）

1. 把 `watermark.html` 存到 iPhone「文件」App。
2. Safari 打开 `devfile.cn`，上传该文件。
3. 拿到 HTTPS 网址，Safari 打开即用；可"分享 → 添加到主屏幕"做成桌面图标。
4. **以后更新**：改完重新上传覆盖即可，桌面图标指向的网址内容随之更新。

## 5. 终端用户使用步骤（可直接发给用户）

1. 用 Safari 打开网址（想要图标就"添加到主屏幕"）。
2. 点中间虚线框，选一张照片或拍一张。
3. 顶部切换"手动输入 / 自动读取"，下方实时预览。
4. 点"保存图片"，长按预览图"存储到照片"，或点"下载图片"。
5. 点"换一张"继续。

> 提示：自动读取需要照片自带拍摄信息；截图 / 转发过的照片可能读不到，会自动切回手动输入。
