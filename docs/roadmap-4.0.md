## 1. 一句话愿景：从"一次处理一张"到"融进你的拍照流程"

3.0 把印迹从"手动加水印的工具"升级成"懂照片、能帮你想的帮手"。但它还停在 **一次处理一张照片** 的节奏上：选一张 → 加水印 / 滤镜 / 贴纸 → 导出 → 换一张。

4.0 想跨过的那条线是：**让印迹融进你真实的拍照流程**——

- 一组照片（活动跟拍 / 一趟旅行 / 一批商品图）要套同一套样式？→ **一次选一组，统一处理、一次导出**，不用一张张重来。
- 想从相册直接处理，懒得"先开网页再选图"？→ **在相册点「分享」就能选「印迹」**，照片直达编辑器。
- 调好了一套喜欢的"水印 + 滤镜 + 色调"组合，想下次复用、发给朋友？→ **打成一个『风格包』，一键套用 / 离线分享**。

一句话：**3.0 让它「懂一张照片」，4.0 让它「处理一整组、并融进你的流程」。** 而且这一切必须延续 3.0 的底线——**全本地、零后端、不破零外链、照片不上传**（第 2 节）。

> 为什么现在能开 4.0：因为 3.0 已经把"单张的样式 / 滤镜 / AI / 自定义 / 分享码"全部打磨成熟。4.0 的"批量 / 风格包"都是在这些**已经能跑、已经验证过**的单张能力之上，做"成组复用"——不是从零造轮子。

## 2. 不可破的底线（沿用 3.0 铁律，4.0 再钉一遍）

> 4.0 任何功能动手前先对这几条。完整版见 `tech.md` 6.11 与 `roadmap-3.0.md` 第 2 节。

1. **照片绝不上传 / 绝不后端处理**：批量也好、分享也好，所有像素运算都在用户设备本地完成。**4.0 绝不引入"把照片传到服务器批处理"这类方案。**
2. **零外链红线延伸到"打包"**：批量导出要把多张图打包，**绝不引第三方 zip / 压缩库**（破单文件 + 零外链）。打包用**自己手写的零依赖 store-zip 编码器**（见第 5 节），或退回逐张导出。
3. **不做账号 / 登录 / 云同步 / 社区后端**：风格包的"分享"**只走离线分享码**（复用 3.0 的 `YINJI1:` 编解码），不做需要服务器的"风格广场 / 云端同步"。
4. **联网 AI 仍默认关 + 当次授权 + 明确告知 + 只发文字**：4.0 的 AI 支线（一句话生成样式 / 地名天气）复用 3.0 的开关 + 授权框架，规矩不变。
5. **PWA 集成不破"打开即用、断网能用"**：分享目标（Web Share Target）是 PWA 原生能力（manifest + sw.js 配置），不引外链、不破单文件；冷启动 / 不支持的平台要优雅降级。

## 3. 已经铺好的地基（4.0 直接往上接，不重做）

4.0 不是平地起高楼，下面这些在 2.x / 3.0 已经备好：

- **多选入口**（v3.0.0-beta.15）：首页 `#fileInput` 已带 `multiple`，`fileInput.change` 按张数分流，`startCollageFromFiles(files)` 已能一次收一组图。批量复用这个入口，只需加"拼图 vs 批量"的分流。
- **单张渲染 / 导出管线**：`getSize()` → `gradedBitmap()`（滤镜）→ 各 `drawXxx`（水印）→ `drawStickerLayer()`（贴纸）→ `exportDataURL(type,q)`。**批量就是"对一组 bitmap 逐张跑这条已验证的管线"**，渲染端零改动。
- **自定义水印 / 滤镜存储 + 分享码**（v2.6.0 / v2.14.0）：`encodeTemplate` / `decodeShareRaw` / `doImport` 已有一套 UTF-8 安全 base64 + `YINJI1:` 前缀 + 按 `type` 路由的编解码。**风格包直接加一个 `type:"yinji-pack"` 分支即可复用整套分享 UI。**
- **主题色**（v3.0.0-beta.5）：`applyAccent()` + `settings.accent`，风格包要打包的"色"已是现成可读可写的状态。
- **PWA**（v2.9.0 起 + v2.11.0 更新提示）：`manifest.webmanifest` + `sw.js` 已就位，分享目标只是给它们加配置 + 加一段 SW 接收逻辑。
- **AI 开关 + 授权框架 + BYOK 通道**（v3.0.0-beta.14 B1）：`settings.aiOnline/aiBase/aiModel/aiKey` + `_aiConsentSession` 当次授权 + `aiEndpoint()` + fetch 封装。4.0 的"一句话生成样式"复用这条通道，只换 prompt 与返回解析。

> 一句话：**批量 = 多选入口（已） + 单张管线（已）逐张跑 + 零依赖打包（新）；风格包 = 分享码体系（已）加一个 type；分享目标 = PWA（已）加配置。** 4.0 的新东西主要是"把已有能力成组化"，加少量工程攻关（store-zip、SW 接收）。

## 4. 三根支柱（4.0 的主体）

### 支柱 1 —— 批量加水印（4.0 头牌，最有感知）

> ✅ **本地版已落地（v3.8.0，2026-06-09）**：南心部署完 v3.x 后直接点名"批量加水印"，提前从 4.0 落地。**入口选择与本规划不同、更省**——没做"首页多选分流 + 批量编辑视图 + 缩略图网格"那套重 UI，而是**复用当前编辑器这套效果**：编辑器保存区加「🗂️ 把这套效果批量加到多张照片」→ 选多张 → 每张套当前 style/滤镜/主题/贴纸，**逐张走单张管线 `renderOneForBatch`（`draw`→`toDataURL`→`bmp.close()` 即时释放）**、`setTimeout` 串行 + 进度条、`BATCH_MAX=30`、**自写零依赖 store-zip（`crc32`+`makeStoreZip`，method 0 不压缩）打包下载**。文字策略 = 尊重当前模式（auto 各读各 EXIF / manual 同文案）。浏览器验证：批 3 张 → zip 合法（EOCD/3 本地头/3 条目）、首条 JPEG、解出底栏色 = 所选样式、编辑器还原、零报错。**4.x 可选增强**：① 缩略图网格预览 + 批量态独立视图（本版用"当前效果"替代，更轻）；② 结果逐张存相册（替 zip，iOS 更友好，待南心反馈）；③ 批量时直接选一个**风格包**套全组；④ 保留原图 EXIF。

- **是什么**：一次选一组照片（如 5～20 张），统一套同一套**水印 + 滤镜 + 主题色**，预览缩略图网格，**一次性全部导出**。水印文字可选"每张读自己的拍摄信息"或"统一一句话"。
- **为什么值得**：活动跟拍、一趟旅行、一批电商图——"成组套同一套样式"是高频刚需，也是把印迹从玩具变生产力工具的关键一步。3.0 的多选入口已证明"一次收一组图"可行。
- **工作量**：大（4.0 的主攻关）。
- **怎么做（技术）**：
  1. **入口分流**：首页多选 2+ 张后，给一个轻量选择"拼成一张（拼图）／每张各自加水印（批量）"。拼图走现有 `startCollageFromFiles`，批量走新 `startBatchFromFiles(files)`。
  2. **批量态**：新增一个"批量编辑"视图——顶部一套样式控件（复用水印 / 滤镜 / 主题色，但作用于"这一组"），中部缩略图网格预览（每张小图实时反映当前样式），底部"全部导出"。
  3. **文字策略**：选 "自动（每张读各自 EXIF）" 或 "统一文字"。自动时每张走 `exifClues` / 自动模式；统一时所有图印同一串。
  4. **导出**：逐张走现有 `getSize()` / `draw()` / `exportDataURL()`，**一张处理完即释放再下一张**（不同时持有 N 个大 canvas，防移动端内存爆）。打包成一个 zip 下载（store 模式，见第 5 节）；不支持下载多文件的环境退回"逐张 Web Share"。
  5. **进度**：N 张逐张处理要给进度条（"3 / 12 张…"），大组别卡死 UI（可 `requestAnimationFrame` / `setTimeout` 切片让出主线程）。
- **依赖**：beta.15 多选入口、单张渲染 / 导出管线、第 5 节 store-zip 编码器。
- **注意**：① roadmap §3.9 早把"批量"标为"重活、确认真有需求再做"——4.0 把它扶正为头牌，因南心选了"成组与流程"。② 移动端内存是真实风险：逐张处理 + 及时释放 + 上限提示（如一次最多 20～30 张）。③ 仍全本地、不上传。④ iOS Safari 批量保存到相册的限制要老实告知（可能要用户配合多次"存储图像"，或导出 zip 让用户自己解压）。
- **验收**：选 5～20 张 → 套一套样式 → 一次导出，**每张水印正确、滤镜一致、自动模式各读各的 EXIF**；大组有进度、不卡死、不爆内存；断网可用。

### 支柱 2 —— PWA 分享目标（从相册一键直达）

- **是什么**：把印迹注册成系统的"分享目标"。在**相册 / 微信 / 任意 App 点「分享」**，列表里就能看到「印迹」，点它 → 照片**直接进印迹编辑器**，不用先开网页再选图。
- **为什么值得**：省掉"打开网页 → 点选图 → 找照片"三步，体验接近原生 App；且是 PWA 标准能力，**不破任何红线**。
- **工作量**：中。
- **怎么做（技术）**：
  1. `manifest.webmanifest` 加 `share_target`：`{ "action": "./share-target", "method": "POST", "enctype": "multipart/form-data", "params": { "files": [{ "name": "photo", "accept": ["image/*"] }] } }`。
  2. `sw.js` 拦截 `POST ./share-target`：从 `formData` 取出图片文件 → 暂存（Cache Storage 放一个临时键，或 IndexedDB）→ `Response.redirect("./watermark.html?shared=1")`。
  3. 页面启动读 `?shared=1` → 向 SW 要刚分享的文件（`postMessage` 握手或读临时缓存）→ `loadFile(thatFile)` 进编辑器 → 用完清临时缓存。
  4. **冷启动**：分享进来时 App 可能没开，靠 SW 接收 + 重定向 + 页面启动握手这条链路兜住。
- **依赖**：现有 `manifest.webmanifest` + `sw.js` + `loadFile`。
- **注意**：① **iOS Safari 对 Web Share Target 支持有限**（Android / 桌面 Chrome 支持好，iOS 历来弱）——必须老实告知南心：这条在安卓 / 桌面体验好，iOS 可能不出现入口，**降级 = 照旧手动选图，不影响任何现有功能**。② manifest / sw 是 PWA 配套文件，不算外链、不破单文件。③ 不在加载期联网。
- **验收**：安卓装 PWA 后，相册"分享"列表出现「印迹」、点了照片进编辑器；iOS 实测表现并在文档诚实标注；不支持的平台手动选图照常。

### 支柱 3 —— 风格包（一套组合一键套用 / 离线分享）

> ✅ **本地版 + 分享均已提前落地（v3.6.0 + v3.7.0，2026-06-08）**：南心连续"继续做下一版"，我把这块从 4.0 提前做了。
> - **本地（v3.6.0）**：「我的风格」在「水印」页顶部，`captureStylePack()` 存 `{style, tone, toneAdj, filterId, accent}`、`applyStylePack()` 一键恢复（`setStyle` 不带 recall + 镜像 recallStyleColor + `applyAccent`），新键 `yinji.stylePacks.v1`。
> - **离线分享 / 导入（v3.7.0）**：`encodePack` / `sharePack` / `importPackObj`，复用 `YINJI1:` + `doImport` 加 `type:"yinji-pack"` 路由。**滤镜免内嵌**（tone+toneAdj 已复现观感、分享码不含 filterId）、**只内嵌自定义水印**（`styleTpl`，导入端自动新建 `custom_*` 并指向；对方没有的水印兜底回落 classic）。浏览器验证：分享码 231 字符、模拟另一台设备导入后风格一字不差 + 套用逐位恢复、自定义水印内嵌导入 customTemplates+1 + 底栏色逐位一致、零报错。
> - **仍待做（4.x）**：**与批量联动**（批量时选一个风格包套全组）——等批量功能落地时一起做。

- **是什么**：把当前的**水印样式 + 滤镜 + 主题色**打成一个『风格包』，存进"我的风格包"；换张照片 / 下次打开**一键套用**整套；还能生成**离线分享码**发给朋友导入。
- **为什么值得**：用户常调出一套自己喜欢的组合，"风格包"让这套组合可复用、可分享，和批量天然搭（批量时直接选一个风格包套全组）；复用 3.0 已有的分享码体系，落地稳。
- **工作量**：中。
- **怎么做（技术）**：
  1. **数据**：`{ type:"yinji-pack", name, style, filter, accent }`——`style` = 水印 id 或自定义水印对象、`filter` = `{tone, toneAdj}` 或自定义滤镜、`accent` = 主题色。存 localStorage 新键 `yinji.packs.v1`（与水印 / 滤镜键分开，避免迁移回归）。
  2. **套用**：`applyPack(pack)` = 一次性 `setStyle` + 设 `state.tone/toneAdj`（或 `filterId`）+ `applyAccent` + `draw()`。
  3. **分享**：编码复用 `SHARE_PREFIX="YINJI1:"` + UTF-8 安全 base64，`doImport` 加 `type==="yinji-pack"` 分支（已有 template / yinji-filter 两条路由，加第三条）。
  4. **UI**："存为风格包" + "我的风格包"列表（套用 / 分享 / 删除）。放哪需与南心确认（设置里？编辑器某处？随机一套旁边？）。
- **依赖**：已有 `encodeTemplate` / `decodeShareRaw` / `doImport`、自定义水印 / 滤镜存储、主题色 `applyAccent`、`state.tone/toneAdj`。
- **注意**：① 别和"自定义水印""自定义滤镜"概念打架——风格包是**更上层的组合**（一个包 = 一套水印 + 滤镜 + 色），UI 要让三者关系清楚、不让用户困惑。② 分享只走离线码，**不做云端风格广场**（要后端，破红线）。③ 套用要可一键还原 / 不强加。
- **验收**：存一个风格包 → 换张照片 → 一键套用 → 水印 + 滤镜 + 主题色全恢复；分享码复制给"另一台"导入正常；与批量联动（批量时选风格包套全组）顺畅。

## 5. 工程化前置（4.0 要先搭好的"地基工程"）

- **零依赖 store-zip 编码器**（批量导出用，**第 2 节红线的关键落地**）：手写一个只做"打包不压缩（store 模式）"的极简 zip 编码器——`CRC32` 查表 + 本地文件头（local file header）+ 中央目录（central directory）+ 结束记录（EOCD），纯字节拼接，约 100 行、零外部库。把 N 张导出的 `Blob`/`Uint8Array` 拼成一个 `.zip` 让用户一次下载。**这是"批量导出又不引外链"的唯一干净解。** 退路：环境不支持多文件下载时退回逐张 Web Share。
- **批量内存管理**：逐张处理 + 处理完即 `canvas.width=0` / 解引用释放；设一次张数上限 + 进度条 + 主线程切片（避免一次性吃满内存 / 卡死）。
- **Web Share Target 接收链路**：SW 拦截 POST → 暂存 → 重定向 → 页面冷启动握手取文件 → 清缓存。要处理"App 没开 / 已开两种情况"和"取完即清"的临时存储生命周期。
- **风格包路由**：把 `doImport` 的 `type` 路由从两类（template / filter）扩到三类（+ pack），保持一个输入框认所有码。

## 6. 里程碑与版本节奏（参考，可调）

4.0 不是一锤子，建议按"感知强 / 风险可控"拆开发、各自先发先验：

- **v4.0.0（头牌）**：**批量加水印**（含 store-zip 编码器、批量态 UI、进度、内存管理）。这是 4.0 最有感知的一版，单独打磨扎实。
- **v4.1**：**风格包**（存 / 套用 / 离线分享，复用分享码），与批量联动（批量套风格包）。
- **v4.2**：**PWA 分享目标**（manifest + SW 接收 + 冷启动握手），安卓 / 桌面先行、iOS 诚实标注降级。
- **AI 支线穿插**（默认关 + 授权，复用 3.0 框架）：**一句话生成样式**（接 B1 的 BYOK 通道，让 AI 返回结构化样式参数 → 落进水印 + 滤镜 + 调色盘）、**地名 / 天气补全**（接 roadmap-3.0 B2，坐标 → 地名 / 天气，授权后联网）。哪版插看价值 / 成本。
- **节奏原则同 2.x / 3.0**：感知强的先发、各自带降级；不攒大招；每发一版照例走完自检 + 双日志 + `md2docx`。

> 与 3.0 / 2.x 的关系：4.0 推进期间，3.0 的零星补丁（端上小料：拍摄地点 / 文字颜色自适应 / 文字微调等，见 `roadmap.md` 第 3 节）仍可按需穿插走 3.0.x，不冲突。

## 7. 风险与验收总纲

- **最大风险：批量导出被迫引外链 / 后端**（破零外链 + 照片不上传两条红线）。对策：**死守"手写 store-zip + 全本地逐张处理"**，绝不引第三方打包库、绝不传服务器；评审时盯死有没有新的 `<script src>` / 上传请求。
- **次大风险：移动端批量内存爆 / 卡死**。对策：逐张处理 + 及时释放 + 张数上限 + 进度切片；真机测大组（20+ 张）表现。
- **平台风险：iOS 对 Web Share Target / 批量保存支持弱**。对策：能力检测 + 优雅降级（不支持就照旧手动选图 / 逐张存），文档诚实标注各平台表现，绝不把"分享目标"做成必需。
- **体验风险：风格包 / 批量把界面搞复杂**。对策：成组能力默认不打扰单张流程；风格包与自定义水印 / 滤镜的关系在 UI 上讲清；批量是"另一条入口"，不挤占单张编辑。
- **隐私验收**：批量 / 分享 / AI 支线上线前，用浏览器网络面板**实测无照片像素外发**；AI 支线默认关、每次授权、只发文字（同 3.0）。

## 8. 4.0 明确不做的（划清边界）

- **不做服务器端批处理 / 云导出**：批量也全本地，照片不上传。
- **不引第三方 zip / 压缩 / 打包库**：手写零依赖 store-zip；守单文件 + 零外链。
- **不做云端风格包社区 / 风格广场 / 云同步**：风格包只走离线分享码（要后端的一概不做，同 `roadmap.md` §6 不做账号 / 云同步）。
- **不为"分享目标"复活原生打包（Capacitor）**：用 PWA 标准能力，iOS 受限就降级，不回头维护两套原生壳。
- **不偏离"加水印"定位做美颜 / 修图 / 视频编辑**：4.0 是"把加水印做成成组流程"，不是变成视频剪辑 / 修图 App（"更广的产物"那条候选方向南心未选，故 4.0 不铺）。
- **AI 支线不设默认开 / 不作前置**：关掉 / 断网，批量 + 风格包 + 单张全部照常。
