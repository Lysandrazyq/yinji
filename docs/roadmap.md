# 印迹 · 更新计划（Roadmap）

> 这份文档回答一个问题：**「印迹」接下来还能更新点什么？**

> 关联文档：现状与已完成功能看 `requirements.md`；技术架构与坑看 `tech.md`（AI 接口与隐私底线在 6.11）；视觉规范看 `design.md`；发布流程看 `steps.md`。

## 1. 每条更新都不能破的底线（5 条）

任何新功能在动手前，都要先对一遍这五条。**破其中任何一条的方案直接否决或改方案**——这些是「印迹」的立身之本，前面踩过坑才定下来的。

1. **单文件 + 零外链**：App 永远是一个自包含的 `watermark.html`，不引入任何需要联网才能加载的资源（外链字体、CDN 脚本、把在线 API 当默认功能）。字体只用系统字体栈。（踩过坑：外链字体在手机内置浏览器里阻塞渲染 → 白屏。）
2. **隐私本地**：照片只在手机本地处理，绝不上传任何服务器。任何"联网"的功能都要单列、默认关闭、并明确告知。
3. **简洁直观**：主界面只暴露必要操作（选图 → 选模式/样式 → 保存）。新功能优先藏进二级面板，不往主界面堆按钮。
4. **PWA 单一上线路径**：核心是网页，改一处全平台同步。不为某个平台另写一套。
5. **改完必收尾**：改完代码必做自检（抽 JS 过 `node --check`、样张验 EXIF、肉眼查排版、grep 查外链）；写两份更新日志；改过的 `.md` 跑 `python tools/md2docx.py` 重生成 `.docx`。

## 3. 3.0 之前：2.x 小步快跑（纯本地打磨，不联网）

> 原则：每条都**不碰红线、纯本地、单款成本低、用户感知强**。攒够一两条就发一个 2.x 小版本，不攒大招。下面大致按"性价比从高到低"排。

### 3.1 继续上新预设模板（性价比最高，随时能做）

- **是什么**：继续扩充内置样式。候选：**春节对联款**、**婚礼 / 喜帖**、**宝宝成长（第 N 天）**、**经纬度坐标款**、**极简编号（1/365）**、**中秋 / 元宵**、更多节气配色。
- **为什么值得**：最稳的"持续上新"，复用现成「调色板 + 共享渲染器」框架，单款成本极低，用户一看就有新鲜感。
- **工作量**：小（每款）。
- **怎么做（技术）**：套现有套路——`TEMPLATES` 加一条登记（id/name/desc/tags）+ `DRAW_DISPATCH` 加一条 `id→draw` 映射 + 归到某个分类（`CAT_IDS`）。能复用 `drawSeasonStyle/drawBrandStyle` 的就传新调色板，不能复用的才写新 draw 函数。
- **注意**：每加一批，分类（`CAT_IDS`/`CAT_OF`）和文档里的"X 款"计数都要同步更新；登记别漏（漏 `DRAW_DISPATCH` 会"搜得到、画不出"）。

### 3.2 文字颜色自适应（自动黑 / 白）

- **是什么**：水印文字根据照片底色自动选黑或白，避免"白字压在白云上看不见"。
- **为什么值得**：解决真实痛点（浅色照片配浅栏、深色照片配深字），提升成片率。
- **工作量**：中。
- **怎么做（技术）**：渲染前用 canvas 采样水印区域照片的平均亮度（缩小取样省性能），亮度高用深字、低用浅字。做成"自动 / 手动"开关，默认自动。纯本地 canvas 运算，不碰红线。
- **注意**：只在"无底栏 / 半透明"样式上收益大（玻璃款、极简右下角）；纯白栏 / 纯黑栏样式本就对比足够，可不参与。

### 3.3 真机适配（横屏 / 系统大字号 / 左右安全区）

- **是什么**：补齐几种真机场景：横屏时不错位、跟随 iPhone「辅助功能-更大字体」、横屏刘海机型左右不被切。（上下安全区已经做了。）
- **为什么值得**：稳健性，避免个别用户"在我手机上歪了 / 被刘海挡了"。
- **工作量**：小-中。
- **怎么做（技术）**：左右补 `env(safe-area-inset-left/right)`；关键文字尺寸用相对单位并测 Dynamic Type；横屏加 `@media (orientation:landscape)` 微调布局。Canvas 水印渲染本身按图片像素算，不受影响，主要是 App 外壳 UI。
- **注意**：不同机型多截几张测；别为横屏大改布局，够用就行。

### 3.4 PWA「有新版本」更新提示 ✅ 已完成（v2.11.0）

- **是什么**：上线新版后，已"装到主屏"的用户能收到"有新版本，点这里刷新"的提示，而不是一直用着缓存里的旧版。
- **为什么值得**：现在靠改 `sw.js` 缓存名强制更新，但用户可能要关掉重开才生效，感知差（这也是现在 v2.9.x 上线后老用户不一定立刻看到新版的原因）。
- **工作量**：中。
- **怎么做（技术）**：service worker `updatefound`/`waiting` 事件 → 页面弹一个轻提示 → 点一下 `skipWaiting` + reload。
- **✅ 已落地（v2.11.0）**：`sw.js` 去掉自动 `skipWaiting` 改 `message` 触发；页面底部液态玻璃提示条 +「刷新」→ `postMessage('SKIP_WAITING')` → `controllerchange` 重载（带 `swReloading` 闸，防首装误刷新）。注意 SW 模型"隔一版生效"：本版上线后老用户需手动刷新一次拿到 v2.11.0，之后的新版才会自动弹提示。
- **注意**：提示要轻（小 toast），别打断正在编辑的用户。

### 3.5 地点信息 —— 只做"安全的那半步"

- **是什么**：在水印上显示拍摄地点。
- **关键取舍**：
  - 从照片 EXIF **读出经纬度** → 纯本地，**不碰红线**，3.0 之前可以做。
  - 把经纬度**转成地名**（"上海·外滩"）→ 要联网查服务，**碰红线**，留到 3.0 及以后当"可选 + 授权"功能（见 4.2）。
- **3.0 之前只做第一步**：读 EXIF GPS → 在水印上显示**坐标**（如 `31.24°N 121.49°E`），并提供一个**"地点"手动输入框**让用户自己填地名。完全本地、零风险。
- **工作量**：小。
- **注意**：GPS 是敏感信息，连"读出来显示"也要让用户能一键关掉，别默认把家庭住址水印上去。

### 3.6 文字微调（字号 / 位置，藏「高级」里）—— 谨慎

- **是什么**：给愿意折腾的用户几个微调项——水印栏高一点 / 矮一点、字大一点 / 小一点、文字左右位置。
- **为什么值得**：不同比例的照片需要不同的水印占比，进阶用户想要掌控感。
- **工作量**：中。
- **怎么做（技术）**：`state` 加 `barScale`/`fontScale`/`align` 等，在 `drawText` 里乘进去；UI 收进一个默认折叠的「高级」抽屉，给"恢复默认"。
- **注意（重要取舍）**：这条**直接挑战"简洁直观"底线**。务必默认隐藏、不出现在主界面。如果会让普通用户困惑，宁可不做。

### 3.7 手动文字多行 / 自选字段

- **是什么**：手动模式支持两行（标题 + 副标题）；自动模式让用户勾选显示哪些字段（如不想显示 ISO）。
- **工作量**：中。
- **怎么做（技术）**：手动文本拆多行渲染；自动模式给字段开关（设置里已有"自动模式显示哪些信息"六项，可在此基础上扩展）。
- **注意**：同样要克制，别让输入区变复杂。

### 3.8 自定义 Logo / 签名图

- **是什么**：让用户上传自己的小 logo 或手写签名图，叠进水印。
- **为什么值得**：摄影师 / 小商家 / 品牌想盖自己的章。
- **工作量**：中-大。
- **怎么做（技术）**：第二个 file input 读成 bitmap → 画进水印区。存储是难点：base64 进 localStorage 会撑爆，建议用 IndexedDB 或仅当次会话内存保留。
- **注意**：纯本地，不碰红线；但要处理透明 PNG、尺寸缩放、位置。

### 3.9 批量加水印（重活，确认真有需求再做）

- **是什么**：一次选多张照片，套同一个模板，批量导出。
- **为什么值得**：旅行整组照片、带货商品图的刚需。
- **工作量**：大。
- **怎么做（技术）**：循环渲染每张 → 逐张保存，或自写一个极简 zip（仅打包不压缩，避免引入压缩库破"零依赖"）。
- **注意**：手机内存是硬约束（多张大图同时在内存里会崩）；要做"逐张处理 + 进度条 + 处理完即释放"。UX 也复杂。

## 4. 3.0 及以后：以 AI 能力为主线

> 3.0 是个分水岭——从"手动加水印工具"升级到"懂照片、能帮你想文案 / 配样式"的工具。**接口已经在 v2.9.1 提前预留好了**，2.x 阶段完全不联网、零 AI 行为，等真要做 AI 时直接往里填即可。

### 4.0 AI 的隐私铁律（先立规矩，再谈功能）

> 这是 AI 部分最重要的一节。**任何 AI 功能上手前先对这四条，破其一直接否决。** 完整版见 `tech.md` 6.11。

1. **照片绝不静默上传**：默认不把照片像素传给任何服务器。预留的 `getContext()` 只读快照**不含照片像素**，只给 AI 看"有没有 EXIF、当前是什么模式 / 样式"这类元信息。
2. **端上优先**：能在手机本地用小模型 / 规则算法做的（如配色、亮度判断），就别联网。
3. **联网 AI 默认关 + 当次授权**：真要用到云端大模型时，必须是用户**明确打开、每次询问、清楚告知"这次会把什么内容联网"**，且默认关闭。
4. **AI 不作核心前置条件**：关掉 AI、断网，App 的全部基础功能（加水印、读 EXIF、导出）必须照常能用。AI 只是锦上添花，不是必需品。

### 4.1 接口已经预留（v2.9.1）

- `window.Yinji.ai`：AI 能力的扩展挂载点（2.x 阶段为空，不做任何事）。
- `window.Yinji.getContext()`：返回一份**只读快照**（当前模式 / 样式 / 是否有 EXIF 等元信息，**不含照片像素、不含 GPS 原始坐标**），将来给 AI 当上下文用。
- 设置页已有「AI · 即将推出」预告段（现在点了只弹一句提示）。

### 4.2 候选 AI 功能（都必须守 4.0 的铁律）

- **智能文案 / 标题建议**：根据 EXIF（设备、时间、参数）+ 用户给的关键词，建议一句水印文案 / 标题。优先纯文本，不传照片。
- **一句话生成样式**：用户说"我想要复古胶片感的水印"，AI 推荐 / 微调一套模板参数（颜色、版式）。
- **场景识别 → 推荐模板**：判断这是风景 / 美食 / 人像 / 街拍，自动推荐合适的预设。**若需要看照片内容，必须走 4.0 第 1、3 条**（默认端上识别；要联网识别则默认关 + 当次授权）。
- **智能配色（端上优先）**：根据照片主色调推荐水印配色。这条**很可能纯本地就能做**（canvas 采样 + 规则），优先不联网实现，归到"端上 AI"。
- **完整调色（接 v2.12.0 的 Beta）**：v2.12.0 已落地"一键套现成色调"的抢先体验版（`getSize()` 单点注入 + 逐像素 `TONE_PARAMS`）。3.0 把它做完整——**强度滑杆、更多胶片色、按样式记忆、可结合上面的"智能配色"让 AI 荐色**。纯端上像素运算，不碰红线。
- **地名 / 天气智能补全**：接 3.5 的"读坐标"，把坐标转成地名、补当天天气。**这条天然联网，必须默认关 + 当次授权 + 明确告知**。

### 4.3 AI 明确不做的（边界）

- **不偷偷传照片**：任何功能都不能在用户没授权时把照片像素发出去。
- **不把 AI 设成默认开 / 强制**：装好 App、断网，一切基础功能照常。
- **不为 AI 引入"必须联网才能加载"的前端依赖**：AI 的联网只发生在"用户授权后的那一次请求"，不能变成打开 App 就要联网（否则又回到白屏老坑）。

## 6. 明确"不打算做"的（划清边界）

写清楚不做什么，和写清楚做什么一样重要——避免以后被带偏。

- **账号 / 登录 / 云同步**：违背"隐私本地 + 零后端"，且对个人项目是巨大维护负担。模板分享已用"离线分享码"解决，够了。
- **滤镜 / 修图 / 美颜**：不是"印迹"的定位（加水印），会让 App 失焦、变重。
- **安卓 / iOS 原生打包（Capacitor）**：v2.3.0 已弃用，不复活（维护两套太重、上架门槛对非程序员太高）。统一走 PWA。
- **HEIC 图像解码（非 Safari 浏览器）**：要引入 libheif 的 WASM（几百 KB），和"单文件轻量零依赖"冲突，**长期搁置**。iPhone Safari 读 HEIC 拍摄信息已正常，这条只影响"非 Safari 看 HEIC 图像本身"。
- **Windows .exe（Tauri）**：已定为**可选、低优先**。PWA 在桌面已能"安装"，价值有限。脚手架在 `cross-platform.md` / `tauri.conf.example.json`，想做时照着走。
- **把"联网查地名 / 天气"做成默认功能**：碰"零外链 + 隐私"红线。只能作为明确可选、每次询问的增强（见 4.2）。
- **任何外链字体 / CDN / 在线 API 当默认依赖**：永久禁止（白屏血泪教训）。
