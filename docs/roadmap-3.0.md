## 1. 一句话愿景：从"加水印的工具"变成"懂这张照片的帮手"

2.x 阶段「印迹」是一把**手动**的工具：你选图、选样式、（要的话）调色，它如实地把水印画上去。所有判断都靠你自己。

3.0 想跨过的那条线是：**让 App 能"看一眼"这张照片、帮你拿主意**——

- 这张照片偏冷还是偏暖？水印文字该用黑还是白才看得清？→ **它替你判断**。
- 想要"复古胶片感"，但不知道该拨哪几个调色滑杆？→ **你说一句话，它把盘拨好**。
- 不知道配什么文案？→ **它根据时间、设备、地点给你两三个候选**。

一句话：**2.x 把"手动调"做到顺手，3.0 在它上面加一层"帮你想"。** 而且这层"帮你想"必须是**锦上添花、可关、可断网**的——这是底线（第 2 节）。

> 为什么现在能开 3.0：因为 2.0 已经把"手动调"的地基打牢了——尤其 v2.12.0 的**一键色调**和 v2.13.0 的**调色盘**（手动拖盘微调冷暖 / 绿品 / 饱和 / 强度）。3.0 的"智能配色 / 完整调色 / 一句话生成样式"全都是在这套**已经能跑、已经验证过**的调色管线上，把"手动拨"换成"帮你拨"——不是从零造轮子。

## 2. 不可破的隐私铁律（这是 3.0 的总闸，破其一直接否决）

> 这是整份 3.0 规划里**最重要的一节**。任何 AI 功能动手前先对这四条。完整版见 `tech.md` 6.11。和 `roadmap.md` 4.0 同源，这里再钉一遍，因为 3.0 每一条都要拿它当尺子量。

1. **照片绝不静默上传**：默认不把照片像素传给任何服务器。预留的 `getContext()` 只读快照**不含照片像素、不含 GPS 原始坐标**，只给 AI 看"有没有 EXIF、当前什么模式 / 样式、照片大概多大、主色调是什么"这类**元信息 / 派生信息**。
2. **端上优先**：能在手机本地用规则算法 / 采样算出来的（配色、亮度判断、按 EXIF 猜场景），就**不联网**。这是 3.0 第一阶段的主力（见 Phase A）。
3. **联网 AI 默认关 + 当次授权 + 明确告知**：真要用云端大模型时，必须用户**亲手打开**、**每次询问**、且**清楚告诉它"这一次会把什么内容发出去"**（例如"只发文字关键词，不发照片"）。默认是关的。
4. **AI 不作核心前置**：关掉 AI、断网，App 的全部基础功能（加水印、读 EXIF、调色盘、导出）必须**照常能用**。AI 只是一层可选的帮手，不是必需品。

> 还有一条工程红线连带着：**AI 不能引入"打开 App 就要联网才能加载"的东西**（外链脚本、启动即下载的大模型）。否则又回到白屏老坑。联网只允许发生在"用户授权后点下去的那一次请求"。这条是 2.x「单文件 + 零外链」红线在 AI 时代的延伸，见第 6 节。

## 3. 已经铺好的地基（不用重做，直接往上接）

3.0 不是平地起高楼，下面这些在 2.x 阶段已经备好了：

- **AI 挂载点 `window.Yinji.ai`**（v2.9.1 预留）：现在是空的，不做任何事。3.0 的 AI 实现往这里填。
- **只读上下文 `window.Yinji.getContext()`**（v2.9.1 预留）：返回当前模式 / 样式 / 是否有 EXIF 等元信息的快照，不含照片像素。3.0 按需扩展它能给出的字段（如"主色调""平均亮度"，见 Phase A）。
- **设置页「AI · 即将推出」预告段**（v2.9.1）：入口已经在设置里占好位，现在点了弹一句提示。3.0 把它变成真正的 AI 开关 + 授权区。
- **一键色调（调色 Beta）**（v2.12.0）：`getSize()` 单点注入 + 逐像素 `TONE_PARAMS`，8 款一键色调，35+ 样式 + 导出零改动全继承。
- **调色盘（手动微调）**（v2.13.0）：在一键色调之上叠一层 `state.toneAdj = {temp, tint, sat, intensity}`——2D 拖盘调冷暖 / 绿品、滑杆调饱和 / 强度。`effectiveParams()` 把"预设 + 盘"合成成一套参数喂给 `applyTone()`。**这套数据结构和管线，就是 3.0「智能配色」要去填的那张表**：AI 只要算出一组 `toneAdj`（或选一个预设 + 微调），调一次 `draw()`，成片就变了——渲染端完全不用动。
- **编辑器「水印 / 调色」分页**（v3.0.0-beta.8）：进编辑器顶部加「水印 / 调色」两个标签，「调色」页现在集中放滤镜 + 调色盘 + 两个端上 AI 帮手（✨ 帮我配色 A1 / ✨ 来点灵感 A3）。**纯 UI 显隐拆分（`setEditTab` 只切 DOM `.hidden`），成图仍同时带水印 + 调色、渲染零改动**；为后续把「AI 总开关 + 当次授权区」放进「调色」页（接 line 50 设置页那段预告）先铺好了版面，AI 帮手从此和滤镜 / 调色盘待在同一处。

> 一句话：**调色盘 v2.13.0 = 智能配色的"手动版"。3.0 的智能配色 = 让 AI / 规则算法去填调色盘的那几个数。** 手动能跑通，自动就只差"谁来填数"。

## 4. 能力地图：分两个阶段，端上先行、联网随后

3.0 的功能分两批做。**Phase A 全部是端上、不联网的**——先把这批做扎实、把"AI 开关 + 授权"的框架打磨好，3.0.0 就发这批。**Phase B 是联网可选增强**——框架稳了再逐条加，每条都默认关、当次授权。

排序原则同 2.x：**不碰红线、用户感知强、单条成本可控**的先做。

### Phase A —— 端上 / 不联网（3.0.0 首发主力）

#### A1. 智能配色（接调色盘，端上规则，**性价比最高、最该先做**）

> ✅ **已落地 v3.0.0-beta.1（2026-06-02）**：调色盘面板顶部「✨ 帮我配色」按钮 → `analyzePhoto()` 把照片缩到 48×48 采样(平均色 / 亮度 / 平均饱和) + `suggestToneAdj()` 规则映射 → 写入 `state.toneAdj` 并经 `syncTonePad()` 同步控件、`draw()` 立即重绘。纯端上、不联网、不下载模型、照片不离开设备;拨完属「未保存的微调」,可一键复位 / 存为滤镜。已注册到 `window.Yinji.ai`(`available` 翻 true)、`getContext()` 增聚合 `colorStats`(非像素)。规则要点:回暖比压冷更积极且都封顶([-22,35]),避免把暖照片洗白。浏览器预览验证通过:冷照回暖(旋钮 67.5%、画面变暖)、暖照封顶压冷(旋钮 39%)、复位还原、手动可接管、无 console 报错。**下一步**:接 A2 的"按样式记忆 + AI 荐色打通"。

- **是什么**：选好照片后，App 看一眼照片的主色调和明暗，**自动建议一组调色**（或在调色盘上把旋钮先帮你拨到一个好看的位置），你可以一键采用、也可以在它的基础上继续手动微调。
- **为什么值得**：直接解决"我也不懂调色，拨半天不知道拨成啥"的痛点；而且它**完全复用 v2.13.0 调色盘的管线**，是 3.0 里离落地最近的一条。
- **工作量**：中。
- **怎么做（技术）**：
  1. 渲染前用 canvas **缩小采样**照片（如缩到 64×64 再读 `getImageData`，省性能），算出主色调（平均色 / 主导色相）、平均亮度、冷暖倾向。
  2. 用**规则 / 启发式**（不是神经网络）把这些映射成一组 `state.toneAdj` 建议值 + 也许一个推荐预设。例如：整体偏冷且暗 → 建议轻微加暖 + 提一点亮度 + 强度 60%。
  3. 把建议灌进**现成的调色盘控件**（`syncTonePad()` 已经能把 `toneAdj` 同步到盘 + 滑杆），用户看到旋钮自己动了、可一键"采用"或"再调"。
  4. 入口：调色盘里加一个「✨ 帮我配色」按钮（默认隐藏在调色盘面板内，不上主界面）。
- **依赖**：v2.13.0 调色盘（已做）；`getContext()` 扩展出"主色调 / 平均亮度"字段。
- **注意**：① 这条**纯本地、不碰红线**，归"端上 AI"。② 一定要"建议"而非"强加"——给一键采用、也给一键还原（调色盘的「复位」已具备）。③ 采样要缩图，别在全分辨率上跑循环卡住手机。
- **验收**：不同色温的样张各试几张，建议值方向合理（暖照片不会被建议更暖到溢出）；采用后能继续手动微调；断网完全可用。

#### A2. 完整调色（把 v2.12.0 的 Beta 收尾成正式版）

- **是什么**：把"调色"从 Beta 升级成完整功能——在已有的一键色调 + 调色盘基础上，再加**更多胶片色预设**、**按样式记忆调色**（不同模板各记各的色）、以及和 A1 智能配色打通（AI 荐色 → 落进调色盘）。
- **为什么值得**：v2.12.0 标的就是"完整版留 3.0"。Beta 已经验证管线没问题，3.0 把广度和记忆补齐，"调色"这块就彻底成熟。
- **工作量**：中。
- **怎么做（技术）**：
  - **更多胶片色**：往 `TONE_PARAMS` 加预设（柯达 / 富士 / 黑白红滤镜等），纯加数据、零新管线。
  - **按样式记忆**：`savePrefs` 现在存的是单组 `tone + toneAdj`；改成"按 `style` 各存一份"（一个 `{styleId: {tone, toneAdj}}` 映射），切样式时取回该样式上次的调色。
  - **AI 荐色打通**：A1 的输出直接写进 `state.toneAdj` 即可，无需额外管线。
- **依赖**：v2.12.0 调色 Beta、v2.13.0 调色盘（都已做）。
- **注意**：① 全程端上像素运算，不碰红线。② "按样式记忆"要给一个"全部清空 / 恢复默认"的出口，别让用户被自己以前的调色困住。③ 预设别堆太多挑花眼，分组折叠。
- **验收**：新胶片色出图正常且随成图保存；切样式能各记各色；和 A1 联动顺畅。

#### A3. 智能文案 / 标题建议（先做"端上规则版"，联网版留 Phase B）

> ✅ **已落地 v3.0.0-beta.7（2026-06-05）**：手动输入文字框下「✨ 来点灵感」按钮 → `suggestCaptions()` 读端上线索生成 2~3 个文案候选,渲染成可点 chip,点一下把文案填进 `#textInput`(`textInput.value=…` → `setMode('manual')` → `draw()`)、可再手改,再点换一批。线索来自:① `exifClues(state.exif)` 从 EXIF 拍摄时间正则解析出**时段**(晨 / 上午 / 午 / 午后 / 暮 / 夜)+ **季节**(春 / 夏 / 秋 / 冬)+ 设备 + 大光圈 / 焦段;② A1 的 `state.toneAdj` 冷暖 / 明暗倾向。句式:按线索拼 4 种结构 + 6 条兜底,去重取前 3。**纯端上、零网络、不读原始像素、不读 GPS、不下载模型**。已注册为 `window.Yinji.ai` 第二个功能(`{id:'caption',label:'文案灵感(端上)',onDevice:true}`);`getContext()` 增只读字段 `hasExif/timeOfDay/season/device/lens`(**仍不含像素、不含原始 GPS**)。浏览器预览验证通过:`ai.list()` = `["smartColor","caption"]`、按钮以 `--accent` 描边、点击出 chip、点 chip 填入并重绘、console 零报错。**下一步**:Phase B 的 B1 联网"一句话生成"在此入口上加"更聪明"的候选(默认关 + 当次授权)。

- **是什么**：根据 EXIF（设备、时间、参数）+ 用户填的一两个关键词，**建议一句水印文案 / 标题**。比如读到"傍晚 + 大光圈 + 50mm" → 建议"暮色·浅焦"之类。
- **为什么值得**：很多人卡在"写啥字"上。哪怕只给几个模板化候选，也能降低动手门槛。
- **工作量**：Phase A 端上版小-中；联网版（见 B1）另算。
- **怎么做（技术）**：
  - **Phase A 端上版**：用**规则 + 词库模板**——把时间段（晨 / 午 / 暮 / 夜）、季节、设备品牌、关键词，套进若干句式模板里生成候选。纯本地、零依赖、零风险。
  - 入口：手动输入框旁一个「✨ 来点灵感」按钮，弹 2~3 个候选，点一下填进去。
- **依赖**：`getContext()` 提供 EXIF 派生信息（时段 / 季节 / 设备）。
- **注意**：① 端上版别吹成"AI 写作"，定位成"灵感模板"更诚实。② 真正的自然语言生成要联网大模型，那是 B1，默认关。
- **验收**：有 EXIF / 无 EXIF 都能给出合理候选；候选可一键采用、可忽略。

#### A4. 场景识别 → 推荐模板（端上启发式优先）

> ✅ **已落地 v3.0.0-beta.18（2026-06-07）**：编辑器「水印」页加「✨ 帮我选样式」按钮 `#sceneRecBtn` → `aiScene()` → `classifyScene()` 用 `analyzeScene(bmp)`（48×48 缩图采样出亮度 / 冷暖 / 偏绿占比 / 偏蓝占比，**聚合不可还原**）+ `exifClues()`（时段 / 季节 / 光圈 / 焦段）按优先级粗判 5 类（夜景 / 风景 / 美食 / 人像 / 日常），`suggestStyles()` 映射到 `SCENE_STYLES` 内置样式 + 有季节则 unshift 当季款，去重取 4，渲染成 `.cap-chip`，点击走现成 `setStyle()`。**纯端上启发式，不联网 / 不识图 / 不下模型 / 不上传照片 / 不读原始 GPS**（守第 2 节铁律，按本条"端上启发式优先、不引 ML 运行时"落地）。注册为 `window.Yinji.ai` 第 4 个 feature（`id:"scene"`、`onDevice:true`，`ai.list()`=`["smartColor","caption","captionOnline","scene"]`）；`getContext()` 增只读 `scene` 字段（仍不含像素 / 原始 GPS）。复用 A3 的按钮 / chip 类，零新 CSS、零核心结构改动；不点按钮手动挑样式照旧（铁律第 4 条）。浏览器预览验证：注入 5 种纯色 photo 经 `ai.run("scene")` 分类逐一命中（绿 / 蓝→风景、暗→夜景、暖亮→美食、中性→日常）、点 chip 套样式（state.style 跟随、`#styleSeg` 高亮）、console 零报错。**至此 Phase A 端上 AI（A1 配色 / A3 文案 / A4 荐样式）全部落地，v3.0.0 beta 全部开发完成。**

- **是什么**：粗略判断这是风景 / 美食 / 人像 / 街拍，**推荐合适的预设样式**（如美食→暖调宝丽来，风景→极简坐标款）。
- **为什么值得**：把 35+ 个样式"挑哪个"的负担减轻，新用户尤其受益。
- **工作量**：中（端上启发式）/ 大（若要真识图）。
- **怎么做（技术）**：
  - **端上启发式（首选）**：先**不真识图**，用能拿到的线索猜——EXIF 的焦段 / 光圈（大光圈近摄 → 可能美食 / 人像）、色彩分布（大面积绿 / 蓝 → 风景）、时间（深夜 → 夜景款）。规则映射到推荐分类。纯本地。
  - **真识图（升级项，谨慎）**：若启发式不够，才考虑端上轻量图像分类——但这需要模型权重，**和"零依赖单文件"冲突**，要走第 6 节的"可选懒加载模块"评估，多半留更后面或干脆不做。
- **依赖**：`getContext()` 暴露 EXIF 派生 + 色彩分布。
- **注意**：① 启发式会猜错，所以是"推荐"不是"自动套用"，用户能无视。② 别为这条引入大模型破红线——拿不准就只做启发式版。
- **验收**：常见几类照片推荐方向大致对；不联网；猜错时不影响正常选样式。

### Phase B —— 联网可选增强（框架稳了再逐条加，全部默认关 + 当次授权）

#### B1. 一句话生成 / 微调样式（联网大模型）

- **是什么**：用户说"我想要复古胶片感、暖一点的水印"，AI 理解后**推荐 / 微调一套模板参数 + 调色盘数值**。
- **为什么值得**：这是 3.0 最"惊艳"的交互，把"挑样式 + 调色"变成"说人话"。
- **工作量**：大。
- **怎么做（技术）**：
  - 把用户那句话 + `getContext()` 的**元信息（不含照片）**发给云端大模型，让它返回一组结构化参数（选哪个预设 / `toneAdj` 怎么拨 / 文案建议），App 落进调色盘 + 样式。
  - 走 `window.Yinji.ai` 挂载点；联网请求只在用户**点了"生成"那一下**发生。
- **依赖**：AI 开关 + 授权框架（第 5 节）；A1/A2 的调色落地能力。
- **注意**：① **默认关、每次询问、明确告知"只发你的文字描述和元信息，不发照片"**。② 要有"不联网也能用"的降级——关掉就退回手动 + A1 端上荐色。③ 网络失败要优雅提示，不卡死。
- **验收**：授权流程清楚；只发约定内容（可在网络面板核对不含图片像素）；断网 / 拒绝授权时 App 照常。

#### B2. 地名 / 天气智能补全（接 roadmap §3.5 的"读坐标"）

- **是什么**：把照片 EXIF 里的 GPS **坐标转成地名**（"上海·外滩"）、补当天**天气**，写进水印。
- **为什么值得**：旅行照"打卡地点 + 天气"是高频需求。
- **工作量**：中。
- **怎么做（技术）**：roadmap §3.5 已说明——"读坐标"是纯本地可先做（2.x 就能上）；"坐标→地名 / 天气"**天然要联网查服务**，归这里。走授权框架，用户点"补全地点"才发起请求。
- **依赖**：roadmap §3.5 的读 GPS 坐标；AI / 联网授权框架（第 5 节）。
- **注意**：① GPS 是最敏感信息，**默认关 + 当次授权 + 明确告知"会把这张照片的坐标发给地图服务"**。② 只发坐标，不发照片。③ 给"只显示坐标、不联网转地名"的纯本地降级（即 §3.5）。
- **验收**：不授权时只显示坐标（本地）；授权后能转地名 / 天气；只发坐标可核对。

## 5. 工程化前置：AI 开关 + 授权框架（Phase A 就要搭好）

不管端上还是联网，3.0 都需要一套**统一的 AI 设置 + 授权机制**。这块是 3.0.0 的"地基工程"，比单个功能更优先：

- **设置页 AI 区**：把现在的「AI · 即将推出」预告段，改成真正的开关组——总开关（默认关）、"端上功能"（配色 / 文案灵感，可单独开，因为不联网所以风险低）、"联网功能"（一句话生成 / 地名天气，默认关且更显眼地警示）。
- **当次授权弹窗**：任何联网 AI 动作前，弹一个说清楚"**这一次**会把**什么内容**发给**谁**、**不会**发什么（照片）"的确认框，用户点"同意这次"才发。不做"以后不再提示"的一劳永逸（或者做也要极其克制）。
- **`getContext()` 扩展**：按 A1/A3/A4 的需要，逐步补充只读字段（主色调、平均亮度、EXIF 派生时段 / 季节 / 设备、色彩分布）——**始终不含照片像素、不含原始 GPS**。
- **`window.Yinji.ai` 实现约定**：定义好端上实现（规则函数）和联网实现（fetch 封装）的统一调用形状，方便分别填充和测试。
- **降级**：每个 AI 入口都要有"AI 关 / 断网"时的表现（隐藏入口或退回手动），保证铁律第 4 条。

> 验收（框架）：总开关关掉后，所有 AI 入口消失、App 与 2.x 体验一致；联网功能默认关；任何联网动作前必有当次授权；授权框里写明"不发照片"。

## 6. 关键技术取舍：怎么在不破"零外链"的前提下做 AI

这是 3.0 最容易出事、必须想清楚的一点：

- **"零外链"红线说的是"打开 App 就要联网加载"的依赖**（外链字体 / CDN 脚本 / 启动即下载模型）——它会阻塞渲染、白屏。**这条永久禁止。**
- **而"用户授权后点一下发起的一次网络请求"不属于此类**——App 本身已经加载完、能离线跑，这次请求是运行时的、可选的、用户触发的。所以**联网 AI（B1/B2）不破红线**，只要：不在加载期发请求、不把云端当默认必需、失败能降级。
- **端上"真神经网络"（TF.js / ONNX / WASM 模型）要慎之又慎**：这些库 + 权重动辄几百 KB 到几 MB，要么打进单文件（爆体积、违背轻量）、要么启动下载（碰红线）。**所以 Phase A 的"端上 AI"一律用规则 / 启发式 / canvas 采样实现，不引入 ML 运行时。** 真有非神经网络解决不了的（如精细识图 A4 升级版），才单独评估"用户主动开启、懒加载的可选模块"，且优先级压到最低、能不做就不做。
- **一句话**：**端上 = 纯 JS 规则（零依赖）；联网 = 用户授权后的一次 fetch（不碰加载期）。两条都不破红线，中间那条"打包大模型进 App"才是要避开的雷。**

## 8. 风险与验收总纲

- **最大风险：隐私铁律被无意破坏**（某条 AI 顺手把照片或坐标发了）。对策：每条联网功能上线前，用浏览器网络面板**实测发出去的请求体**，确认不含照片像素 / 原始 GPS；写进发布自检清单。
- **次大风险：为 AI 引入加载期依赖**（破"零外链"白屏老坑）。对策：第 6 节那条尺子——端上只用纯 JS 规则，联网只在用户授权后 fetch；评审时盯死有没有新的 `<script src>` / 启动下载。
- **体验风险：AI 变成"必须用"**。对策：铁律第 4 条——关掉 AI / 断网，加水印 / EXIF / 调色盘 / 导出全部照常；每版都测一遍"全关"状态。
- **质量风险：建议不靠谱反而帮倒忙**。对策：所有 AI 输出都是"建议"，必有一键还原 / 忽略；端上规则先用样张调参到方向稳定再上。

## 9. 3.0 明确不做的（划清边界，和 roadmap §6 一致）

- **不偷偷传照片 / 坐标**：任何功能在用户没授权时都不发照片像素、不发原始 GPS。
- **不把 AI 设成默认开 / 强制前置**：装好断网，基础功能照常。
- **不为 AI 引入"打开就要联网加载"的前端依赖**：联网只发生在用户授权后的那一次请求。
- **不打包大模型进单文件 / 不启动下载模型**：端上 AI 用规则实现；真要 ML 运行时只能作"用户主动开启的懒加载可选模块"，优先级最低、能不做就不做。
- **不做账号 / 登录 / 云同步**（同 roadmap §6）：和隐私本地 + 零后端冲突。
- **不做滤镜 / 修图 / 美颜**（同 roadmap §6）：偏离"加水印"定位。AI 调色只服务"水印成片观感"，不做人脸 / 磨皮这类修图。
